# gstack

> "لا أعتقد أنني كتبت سطرًا من الكود ربما منذ ديسمبر، بشكل أساسي، وهو تغيير كبير للغاية." — [أندريه كارباثي](https://fortune.com/2026/03/21/andrej-karpathy-openai-cofounder-ai-agents-coding-state-of-psychosis-openclaw/)، بودكاست No Priors، مارس 2026

عندما سمعت كارباثي يقول هذا، أردت أن أكتشف كيف. كيف يمكن لشخص واحد أن يُنتج مثل فريق من عشرين؟ بيتر شتاينبرغر بنى [OpenClaw](https://github.com/openclaw/openclaw) — 247 ألف نجمة على GitHub — بمفرده بشكل أساسي مع وكلاء الذكاء الاصطناعي. الثورة هنا. باني واحد مع الأدوات الصحيحة يمكنه التحرك أسرع من فريق تقليدي.

أنا [غاري تان](https://x.com/garrytan)، رئيس ومدير تنفيذي لـ [Y Combinator](https://www.ycombinator.com/). عملت مع آلاف الشركات الناشئة — Coinbase، Instacart، Rippling — عندما كانوا شخصًا أو اثنين في مرآب. قبل YC، كنت أحد أوائل المهندسين/مديري المنتجات/المصممين في Palantir، شاركت في تأسيس Posterous (بيعت لـ Twitter)، وبنيت Bookface، الشبكة الاجتماعية الداخلية لـ YC.

**gstack هي إجابتي.** أبني منتجات منذ عشرين عامًا، والآن أُنتج كودًا أكثر من أي وقت مضى. في آخر 60 يومًا: **أكثر من 600,000 سطر من كود الإنتاج** (35% اختبارات)، **10,000-20,000 سطر في اليوم**، بدوام جزئي، بينما أدير YC بدوام كامل. هذا آخر `/retro` لي عبر 3 مشاريع: **140,751 سطرًا مُضافًا، 362 commit، ~115 ألف سطر صافي** في أسبوع واحد.

نفس الشخص. عصر مختلف. الاختلاف هو في الأدوات.

**gstack هي كيف أفعل ذلك.** يحول Claude Code إلى فريق هندسة افتراضي — مدير تنفيذي يعيد التفكير في المنتج، مدير هندسة يُثبّت البنية، مصمم يكتشف محتوى الذكاء الاصطناعي الرديء، مراجع يجد أخطاء الإنتاج، قائد QA يفتح متصفحًا حقيقيًا، مسؤول أمان يُشغّل مراجعات OWASP + STRIDE، ومهندس إصدار يُطلق الـ PR. عشرون متخصصًا وثمانية أدوات قوية، كلها أوامر شرطة مائلة، كلها Markdown، كلها مجانية، رخصة MIT.

هذا هو مصنع البرمجيات مفتوحة المصدر الخاص بي. أستخدمه كل يوم. أشاركه لأن هذه الأدوات يجب أن تكون متاحة للجميع.

انسخه. حسّنه. اجعله خاصًا بك. وإذا أردت انتقاد البرمجيات مفتوحة المصدر المجانية — فأهلاً بك، لكنني أفضل أن تجربها أولاً.

**لمن هذا:**
- **المؤسسون والرؤساء التنفيذيون** — خاصة التقنيين الذين لا يزالون يريدون الإنتاج
- **مستخدمو Claude Code لأول مرة** — أدوار منظمة بدلاً من مطالبة فارغة
- **قادة التقنية ومهندسو staff** — مراجعة دقيقة، QA، وأتمتة الإصدار في كل PR

## البداية السريعة

1. ثبّت gstack (30 ثانية — انظر أدناه)
2. شغّل `/office-hours` — اصف ما تبنيه
3. شغّل `/plan-ceo-review` على أي فكرة ميزة
4. شغّل `/review` على أي فرع به تغييرات
5. شغّل `/qa` على رابط staging الخاص بك
6. توقف هناك. ستعرف إذا كان هذا مناسبًا لك.

## التثبيت — 30 ثانية

**المتطلبات:** [Claude Code](https://docs.anthropic.com/en/docs/claude-code)، [Git](https://git-scm.com/)، [Bun](https://bun.sh/) v1.0+، [Node.js](https://nodejs.org/) (Windows فقط)

### الخطوة 1: التثبيت على جهازك

افتح Claude Code والصق هذا. Claude يفعل الباقي.

> تثبيت gstack: شغّل **`git clone https://github.com/garrytan/gstack.git ~/.claude/skills/gstack && cd ~/.claude/skills/gstack && ./setup`** ثم أضف قسم "gstack" إلى CLAUDE.md يقول استخدام مهارة /browse من gstack لجميع تصفح الويب، لا تستخدم أبدًا أدوات mcp__claude-in-chrome__*، ويسرد المهارات المتاحة: /office-hours، /plan-ceo-review، /plan-eng-review، /plan-design-review، /design-consultation، /review، /ship، /land-and-deploy، /canary، /benchmark، /browse، /qa، /qa-only، /design-review، /setup-browser-cookies، /setup-deploy، /retro، /investigate، /document-release، /codex، /cso، /autoplan، /careful، /freeze، /guard، /unfreeze، /gstack-upgrade. ثم اسأل المستخدم إذا كان يريد أيضًا إضافة gstack إلى المشروع الحالي ليحصل عليه زملاء الفريق.

### الخطوة 2: الإضافة إلى المستودع الخاص بك ليحصل عليه زملاء الفريق (اختياري)

> إضافة gstack إلى هذا المشروع: شغّل **`cp -Rf ~/.claude/skills/gstack .claude/skills/gstack && rm -rf .claude/skills/gstack/.git && cd .claude/skills/gstack && ./setup`** ثم أضف قسم "gstack" إلى CLAUDE.md لهذا المشروع الذي يقول استخدام مهارة /browse من gstack لجميع تصفح الويب، لا تستخدم أبدًا أدوات mcp__claude-in-chrome__*، يسرد المهارات المتاحة: /office-hours، /plan-ceo-review، /plan-eng-review، /plan-design-review، /design-consultation، /review، /ship، /land-and-deploy، /canary، /benchmark، /browse، /qa، /qa-only، /design-review، /setup-browser-cookies، /setup-deploy، /retro، /investigate، /document-release، /codex، /cso، /careful، /freeze، /guard، /unfreeze، /gstack-upgrade، ويخبر Claude أنه إذا لم تعمل مهارات gstack، شغّل `cd .claude/skills/gstack && ./setup` لبناء الملف الثنائي وتسجيل المهارات.

الملفات الحقيقية تُcommit في المستودع الخاص بك (ليس submodule)، لذا `git clone` يعمل ببساطة. كل شيء يعيش داخل `.claude/`. لا شيء يلمس PATH الخاص بك أو يعمل في الخلفية.

### Codex أو Gemini CLI أو Cursor

gstack يعمل على أي وكيل يدعم [معيار SKILL.md](https://github.com/anthropics/claude-code). المهارات تعيش في `.agents/skills/` ويتم اكتشافها تلقائيًا.

```bash
git clone https://github.com/garrytan/gstack.git ~/.codex/skills/gstack
cd ~/.codex/skills/gstack && ./setup --host codex
```

أو اترك setup يكتشف تلقائيًا أي الوكلاء لديك مثبتين:

```bash
git clone https://github.com/garrytan/gstack.git ~/gstack
cd ~/gstack && ./setup --host auto
```

هذا يُثبّت في `~/.claude/skills/gstack` و/أو `~/.codex/skills/gstack` اعتمادًا على المتاح. جميع المهارات الـ 28 تعمل عبر جميع الوكلاء المدعومين. مهارات الأمان القائمة على الخطافات (careful، freeze، guard) تستخدم نصوص أمان مضمنة على مضيفات غير Claude.

## الـ sprint

gstack هو عملية، ليست مجموعة أدوات. المهارات تُشغّل بالترتيب الذي يُشغّل به الـ sprint:

**التفكير → التخطيط → البناء → المراجعة → الاختبار → الإصدار → التأمل**

كل مهارة تُغذي التالية. `/office-hours` تكتب مستند تصميم يقرأه `/plan-ceo-review`. `/plan-eng-review` تكتب خطة اختبار يلتقطها `/qa`. `/review` يلتقط أخطاء يتحقق `/ship` من إصلاحها. لا شيء يسقط عبر الشقوق لأن كل خطوة تعرف ما جاء قبلها.

| المهارة | متخصصك | ما يفعله |
|-------|----------------|--------------|
| `/office-hours` | **YC Office Hours** | ابدأ هنا. ستة أسئلة إلزامية تعيد تأطير منتجك قبل كتابة الكود. تعارض تأطيرك، تتحدى المقدمات، تولّد بدائل تنفيذ. مستند التصميم يُغذي كل مهارة لاحقة. |
| `/plan-ceo-review` | **الرئيس التنفيذي / المؤسس** | إعادة التفكير في المشكلة. إيجاد المنتج 10 نجوم المختبئ داخل الطلب. أربعة أوضاع: التوسع، التوسع الانتقائي، الحفاظ على النطاق، التخفيض. |
| `/plan-eng-review` | **مدير الهندسة** | تثبيت البنية، تدفق البيانات، المخططات، حالات الحافة، والاختبارات. يجبر الافتراضات المخفية على الظهور. |
| `/plan-design-review` | **مصمم أول** | يقيّم كل بُعد تصميم 0-10، يشرح كيف يبدو الـ 10، ثم يعدّل الخطة للوصول إلى هناك. اكتشاف محتوى الذكاء الاصطناعي الرديء. تفاعلي — سؤال AskUserQuestion واحد لكل خيار تصميم. |
| `/design-consultation` | **شريك التصميم** | بناء نظام تصميم كامل من الصفر. يبحث المشهد، يقترح مخاطر إبداعية، يولد نماذج منتج واقعية. |
| `/review` | **مهندس staff** | يجد الأخطاء التي تمر بـ CI لكنها تنفجر في الإنتاج. يُصلح تلقائيًا الواضحة. يعلّم فجوات الاكتمال. |
| `/investigate` | **مكتشف الأخطاء** | تصحيح منهجي للسبب الجذري. قانون حديدي: لا إصلاحات بدون تحقيق. يتتبع تدفق البيانات، يختبر الفرضيات، يتوقف بعد 3 إصلاحات فاشلة. |
| `/design-review` | **مصمم يبرمج** | نفس تدقيق /plan-design-review، ثم يُصلح ما يجده. commits ذرية، لقطات قبل/بعد. |
| `/qa` | **قائد QA** | يختبر تطبيقك، يجد أخطاء، يُصلحها بـ commits ذرية، يعيد التحقق. يولد تلقائيًا اختبارات انحدار لكل إصلاح. |
| `/qa-only` | **مراسل QA** | نفس منهجية /qa لكن تقرير فقط. تقرير خطأ نقي بدون تغييرات كود. |
| `/cso` | **كبير مسؤولي الأمان** | OWASP Top 10 + نموذج تهديد STRIDE. ضوضاء صفرية: 17 استبعادًا للإيجابيات الكاذبة، بوابة ثقة 8/10+، تحقق مستقل من النتائج. كل نتيجة تتضمن سيناريو استغلال ملموس. |
| `/ship` | **مهندس الإصدار** | مزامنة main، تشغيل الاختبارات، تدقيق التغطية، push، فتح PR. يُنشئ أطر اختبار إذا لم يكن لديك واحد. |
| `/land-and-deploy` | **مهندس الإصدار** | دمج الـ PR، انتظار CI والنشر، التحقق من صحة الإنتاج. أمر واحد من "موافق عليه" إلى "تم التحقق في الإنتاج". |
| `/canary` | **SRE** | حلقة مراقبة ما بعد النشر. يراقب أخطاء وحدة التحكم، تراجعات الأداء، وفشل الصفحة. |
| `/benchmark` | **مهندس الأداء** | خط أساس لأوقات تحميل الصفحة، Core Web Vitals، وأحجام الموارد. يقارن قبل/بعد في كل PR. |
| `/document-release` | **كاتب تقني** | تحديث كل وثائق المشروع لتطابق ما أطلقته للتو. يلتقط READMEs القديمة تلقائيًا. |
| `/retro` | **مدير الهندسة** | retro أسبوعي واعٍ للفريق. تفصيلات لكل شخص، سلاسل الإصدار، اتجاهات صحة الاختبار، فرص النمو. `/retro global` يعمل عبر جميع مشاريعك وأدوات الذكاء الاصطناعي (Claude Code، Codex، Gemini). |
| `/browse` | **مهندس QA** | متصفح Chromium حقيقي، نقرات حقيقية، لقطات حقيقية. ~100ms لكل أمر. |
| `/setup-browser-cookies` | **مدير الجلسة** | يستورد ملفات تعريف الارتباط من متصفحك الحقيقي (Chrome، Arc، Brave، Edge) إلى الجلسة headless. يختبر الصفحات المصادق عليها. |
| `/autoplan` | **خط أنابيب المراجعة** | أمر واحد، خطة تمت مراجعتها بالكامل. يُشغّل تلقائيًا مراجعة الرئيس التنفيذي → التصميم → الهندسة مع مبادئ قرار مشفرة. يُظهر فقط قرارات الذوق لموافقتك. |

### أدوات الطاقة

| المهارة | ما تفعله |
|-------|-------------|
| `/codex` | **الرأي الثاني** — مراجعة كود مستقلة من OpenAI Codex CLI. ثلاثة أوضاع: مراجعة (بوابة نجاح/فشل)، تحدي خصمي، واستشارة مفتوحة. تحليل عبر النماذج عندما يعمل كل من `/review` و `/codex`. |
| `/careful` | **حواجز الأمان** — يحذر قبل الأوامر المدمرة (rm -rf، DROP TABLE، force-push). قل "be careful" للتفعيل. تجاوز أي تحذير. |
| `/freeze` | **قفل التحرير** — يقيد تحرير الملفات إلى دليل واحد. يمنع التغييرات العرضية خارج النطاق أثناء التصحيح. |
| `/guard` | **الأمان الكامل** — `/careful` + `/freeze` في أمر واحد. أقصى أمان لعمل الإنتاج. |
| `/unfreeze` | **إلغاء القفل** — إزالة حدود `/freeze`. |
| `/setup-deploy` | **مُهيئ النشر** — إعداد لمرة واحدة لـ `/land-and-deploy`. يكتشف منصتك، رابط الإنتاج، وأوامر النشر. |
| `/gstack-upgrade` | **المُحدّث الذاتي** — ترقي gstack إلى الأحدث. يكتشف التثبيت العام أو المدمج، يزامن كليهما، يُظهر ما تغير. |

## Sprints متوازية

gstack يعمل بشكل جيد مع sprint واحد. يصبح مثيرًا للاهتمام مع عشرة تعمل في وقت واحد.

[Conductor](https://conductor.build) يُشغّل جلسات Claude Code متعددة بالتوازي — كل واحدة في مساحة عمل معزولة خاصة بها. جلسة على `/office-hours`، أخرى على `/review`، ثالثة تُنفذ ميزة، رابعة تُشغّل `/qa`. كلها في نفس الوقت. بنية الـ sprint هي ما يجعل التوازي يعمل — بدون عملية، عشرة وكلاء هم عشرة مصادر فوضى. مع عملية، كل وكيل يعرف بالضبط ماذا يفعل ومتى يتوقف.

## الخصوصية والقياس عن بُعد

يتضمن gstack قياس استخدام **اختياري** للمساعدة في تحسين المشروع. إليك ما يحدث بالضبط:

- **معطل افتراضيًا.** لا يُرسل أي شيء إلى أي مكان ما لم تقل نعم صراحة.
- **عند التشغيل الأول،** يسأل gstack إذا كنت تريد مشاركة بيانات استخدام مجهولة. يمكنك قول لا.
- **ما يُرسل (إذا اخترت المشاركة):** اسم المهارة، المدة، النجاح/الفشل، إصدار gstack، نظام التشغيل. هذا كل شيء.
- **ما لا يُرسل أبدًا:** الكود، مسارات الملفات، أسماء المستودعات، أسماء الفروع، المطالبات، أو أي محتوى منشأ من المستخدم.
- **التغيير في أي وقت:** `gstack-config set telemetry off` يعطل كل شيء فورًا.

تُخزن البيانات في [Supabase](https://supabase.com) (بديل مفتوح المصدر لـ Firebase). المخطط في [`supabase/migrations/001_telemetry.sql`](supabase/migrations/001_telemetry.sql) — يمكنك التحقق بالضبط مما يُجمع. مفتاح Supabase القابل للنشر في المستودع هو مفتاح عام (مثل مفتاح API لـ Firebase) — سياسات الأمان على مستوى الصف تقيده للوصول للإدراج فقط.

**التحليلات المحلية متاحة دائمًا.** شغّل `gstack-analytics` لرؤية لوحة استخدامك الشخصية من ملف JSONL المحلي — لا حاجة لبيانات بعيدة.

## استكشاف الأخطاء

**المهارة لا تظهر؟** `cd ~/.claude/skills/gstack && ./setup`

**`/browse` يفشل؟** `cd ~/.claude/skills/gstack && bun install && bun run build`

**تثبيت قديم؟** شغّل `/gstack-upgrade` — أو عيّن `auto_upgrade: true` في `~/.gstack/config.yaml`

**مستخدمو Windows:** يعمل gstack على Windows 11 عبر Git Bash أو WSL. Node.js مطلوب بالإضافة إلى Bun — لدى Bun خطأ معروف مع نقل pipe لـ Playwright على Windows ([bun#4253](https://github.com/oven-sh/bun/issues/4253)). خادم التصفح يعود تلقائيًا إلى Node.js. تأكد من أن كلاً من `bun` و `node` في PATH الخاص بك.
